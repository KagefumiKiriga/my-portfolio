# 2026/07/17 開発日記：CrowdWorks案件分岐＆ナレッジ自動保存ワークフロー完成

## 今日やったこと

CrowdWorksの案件応募を自動化するDifyワークフローを、シンプルな「入力→LLM→出力」の3段構成から、
条件分岐＋自動採点＋ナレッジ自動保存を備えた本格的な仕組みにアップグレードした。

## 完成したワークフローの全体像

案件文を入力
  ↓
判定LLM（稼働条件マッチ度をlevel 1〜3で判定）
  ↓ level = 3 のみ通過
提案文LLM（応募文を自動生成、自己採点スコアつき）
  ↓ score ≧ 80 のみ通過
HTTPリクエスト（Difyナレッジベースに提案文を自動保存）
  ↓
回答（画面に提案文を表示）

- level ≠ 3 の場合 → 見送りコメントLLMが理由を一言添えて終了
- score < 80 の場合 → 保存せずそのまま回答して終了

## 途中でハマった問題と学び

### 1. ノードのリネームで参照が壊れる
LLMノードを「判定LLM」「提案文LLM」などにリネームした際、既存の「回答」ノードが古い名前（LLM）
への参照を持ったままになり、「無効な変数です」エラーが発生。
→ 教訓：ノードをリネームしたら、それを参照している下流のノードも必ず確認する。

### 2. 構造化出力の入れ子構造をIF/ELSEが正しく読めない
LLM2 / structured_output / level のような入れ子（オブジェクトの中のオブジェクト）を
IF/ELSEの条件式に直接指定すると、「Invalid actual value type: string or boolean」エラーが発生。
→ 教訓：入れ子構造の値をIF/ELSEで使う場合は、間に「コード実行」ノードを挟んで
単純な数値・真偽値として取り出してから比較する。この構成をワンパターン化しておくと今後も応用できる。

### 3. AIがプロンプト内の「お手本文言」をそのまま丸暗記する
判定基準プロンプトの中に「reasonに『時間条件の記載なし』と明記すること」という例文を書いたところ、
AIがその文言をあらゆる状況でコピペするようになってしまい、正しい判定ができなくなった。
→ 教訓：プロンプトには「使うべき固定フレーズ」ではなく「判断のロジック」を書く。
最優先ルールを明示し、誤りやすいフレーズは名指しで禁止すると効果的。

### 4. DifyのAPIキーは種類が複数ある
ナレッジベースへの自動保存でHTTPリクエストがずっと401エラーになっていた原因は、
「アプリ用（app-）」のAPIキーを使っていたこと。正しくは「サービスAPI（dataset-）」のキーが必要だった。
→ 教訓：Difyには複数の種類のAPIキーがあり、用途によって発行場所が異なる。

### 5. JSONに変数を挿入するときは、必ずカーソル位置を意識する
既存の文字列を選択した状態で変数チップを挿入すると、意図しない位置に文字が混入し、
JSON構造が壊れて「indexing_technique is required」エラーが発生。
→ 教訓：変数を挿入するときは、空のカギカッコの中にピンポイントでカーソルを置いてから{を入力する。

## 次にやりたいこと（積み残し）

- 過去の高評価応募文（ナレッジに溜まったデータ）を、判定LLMや提案文LLMの参考情報として
読み込ませるRAG構成への拡張
- テスト実行で溜まった検証用データをナレッジベースから削除し、本番運用をスタート

## 感想

配線ミス・型の不一致・APIキーの取り違えなど、地味なトラブルシューティングの連続だったが、
1つずつ原因を切り分けていくことで、最終的に「良い案件だけを自動で見極めて、良い提案文だけを
自動でアーカイブする」という当初のゴールまで到達できた。特に「入れ子構造はコード実行ノードで
平坦化する」というパターンは、今後別のワークフローを作るときにもすぐ使える汎用テクニックとして
身についたのが大きな収穫。
